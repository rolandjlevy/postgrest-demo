# PostgREST demo

A working PostgREST API with a Next.js playground to poke at it.

**The point of this project:** there are no route handlers. Not one. Every endpoint
is generated by PostgREST from `db/init.sql`. The schema *is* the API.

---

## Run it

You need Docker and Node.

```bash
# 1. Start Postgres + PostgREST (init.sql runs automatically on first boot)
docker compose up -d

# 2. Start the playground UI
cd web
npm install
npm run dev
```

- API → http://localhost:3000
- Playground → http://localhost:3001

Sanity check before opening the UI:

```bash
curl http://localhost:3000/branches
```

If you change `db/init.sql`, rebuild from scratch — the init script only runs on
first boot of an empty volume:

```bash
docker compose down -v && docker compose up -d
```

## The four ideas worth taking away

**1. Tables, views, and functions become URLs automatically.**

| In the database | Over HTTP |
|---|---|
| `api.branches` (table) | `GET /branches` |
| `api.public_branches` (view) | `GET /public_branches` |
| `api.branches_near()` (function) | `POST /rpc/branches_near` |

**2. The querystring is the query.** Filters map onto SQL directly:

```
GET /branches?select=name,lat&lat=gt.52.5&order=lat.desc&limit=4
```

```sql
SELECT name, lat FROM api.branches
WHERE lat > 52.5 ORDER BY lat DESC LIMIT 4;
```

Nobody wrote an endpoint for that combination. It just works. Operators:
`eq`, `gt`, `lt`, `gte`, `lte`, `like`, `in`, `is`.

**3. Authorization lives in Postgres, not in application code.**

`authenticator` is a deliberately powerless role — it can log in but owns no data
access (`NOINHERIT` matters here). Per request, PostgREST *switches* into
`web_anon` (unauthenticated) or `web_user` (authenticated).

Row Level Security then decides what each role can see:

```sql
CREATE POLICY branches_anon_select ON api.branches
  FOR SELECT TO web_anon
  USING (is_active = true AND show_on_b2c = true);
```

Verified behaviour: `web_anon` sees **7 branches**, `web_user` sees all **10**.
The three internal depots aren't filtered out by the API — the database
physically refuses to return them. No malformed query, injection, or forgotten
`WHERE` clause in application code can bypass that. It fails **closed**.

Compare that to a conditional `WHERE` assembled in application code, which fails
**open** if the app forgets to apply it.

**4. Grants can be asymmetric, and that's useful.**

`web_anon` has `INSERT` on `api.enquiries` but no `SELECT`. So anyone can submit
an enquiry, and nobody can read enquiries back out. That's a public contact form,
enforced at the database. Try the "Read enquiries (denied)" endpoint in the
playground — the 401 is the lesson, not a bug.

## Try these

```bash
# Filtering
curl "http://localhost:3000/branches?postcode=eq.LE1%201AA"

# Composing
curl "http://localhost:3000/branches?select=name,lat&lat=gt.52.5&order=lat.desc"

# The RPC — haversine distance, zero application code
curl -X POST http://localhost:3000/rpc/branches_near \
  -H "Content-Type: application/json" \
  -d '{"lat": 52.6369, "lon": -1.1398, "radius_miles": 40}'

# Insert works…
curl -X POST http://localhost:3000/enquiries \
  -H "Content-Type: application/json" \
  -d '{"branch_id":1,"name":"Ada","email":"ada@example.com","message":"M6 bolts?"}'

# …but reading back does not (401 — by design)
curl http://localhost:3000/enquiries

# PostgREST also self-documents: OpenAPI spec of every generated endpoint
curl http://localhost:3000
```

## Where this fits, honestly

**Good fit:** internal tools, admin panels, prototypes, read-heavy APIs — anywhere
the data model *is* the API and you want to skip CRUD boilerplate. It genuinely
eliminates the tedious 80% of REST endpoints.

**Poor fit:** complex orchestration (call three services, transform, then write),
heavy business logic, or workflows that don't map cleanly onto database
operations. You *can* push that into PL/pgSQL functions, but significant business
logic in PL/pgSQL is usually worse than in TypeScript — harder to test, harder to
debug, harder to hire for.

## Next step

JWT auth. PostgREST reads a `role` claim from a JWT and switches into that
Postgres role for the request — which is how `web_user` gets used in practice.
Set `PGRST_JWT_SECRET` and send `Authorization: Bearer <token>`.
