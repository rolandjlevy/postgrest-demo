# PostgREST demo

A working PostgREST API with a Next.js playground to poke at it.

**The point of this project:** there are no route handlers. Not one. Every endpoint
is generated by PostgREST from `db/init.sql`. The schema *is* the API.

---

## Quick start

Pick your environment:

- **[Local (Docker Desktop)](#running-locally)** — fastest if you already have Docker running.
- **[GitHub Codespaces](#running-in-codespaces)** — nothing to install locally, but has three gotchas worth knowing before you start.

Once running:
- API → http://localhost:3000 (or the Codespaces forwarded URL)
- Playground → http://localhost:3001 (or the Codespaces forwarded URL)

---

## Running locally

You need Docker Desktop running, and Node.js 18+.

```bash
# 1. Start Postgres + PostgREST
docker compose up -d

# 2. Verify PostgREST is reachable — should return 7 branches as JSON
curl http://localhost:3000/branches

# 3. Start the Next.js playground
cd web
npm install
npm run dev
```

Open http://localhost:3001 in your browser. That's it.

## Running in Codespaces

Codespaces adds three steps that local doesn't need. Skipping any one will make the playground fail with `Failed to fetch`.

### Before opening the Codespace

Make sure `.devcontainer/devcontainer.json` includes `"moby": false` in the docker-in-docker feature options. The default fails on Debian trixie (which current Codespaces images use). If you cloned this repo, it's already set correctly — just noting why.

### After the Codespace starts

**1. Start the containers.**

```bash
docker compose up -d
docker compose ps                     # both containers should show "Up"
curl http://localhost:3000/branches   # should return 7 branches
```

If `docker compose ps` shows either container as `Exited` or nothing at all, run `docker compose logs db --tail=50` and see the [Troubleshooting](#troubleshooting) section — the most common cause is the `init.sql` bind-mount issue.

**2. Make port 3000 public.**

The Next.js playground runs in *your* browser (not inside the Codespace), so it needs public access to the PostgREST URL. Without this step, the playground shows "Failed to fetch" no matter what.

- Bottom of VS Code → **Ports** tab
- Right-click the row for port **3000** → **Port Visibility** → **Public**

Port 3001 can stay Private — Codespaces handles that via authenticated forwarding.

**3. Point the playground at the forwarded URL.**

Copy the forwarded URL for port 3000 from the Ports panel (it looks like `https://your-codespace-3000.app.github.dev`), then:

```bash
cd web
POSTGREST_URL=https://your-codespace-3000.app.github.dev npm run dev
```

No trailing slash on the URL.

Then, back in the Ports panel, click the globe icon next to port 3001 to open the playground.

## Troubleshooting

### `container postgrest_demo_db exited (1)`

Almost certainly the bind-mount trap. Check what Postgres said:

```bash
docker compose logs db --tail=30
```

If you see `could not read from input file: Is a directory`, the file `db/init.sql` doesn't exist as a file — Docker created it as an empty root-owned directory on first `up`. To recover:

```bash
docker compose down -v
sudo rm -rf db/init.sql
sudo chown -R $USER:$USER db/     # if the parent got root-owned too
ls -la db/init.sql                # should not exist at this point
```

Then check that the real `init.sql` file is present in your working tree. If it's missing, restore it from the repo (`git checkout db/init.sql`) or copy it in. Verify:

```bash
ls -la db/init.sql
# expected: regular file (starts with -), owned by you, ~9 KB
```

**Only then** run `docker compose up -d` again. Doing an `up` before the real file is in place puts you back at square one.

### `Failed to fetch` in the playground

The browser can't reach PostgREST. Check port 3000's visibility in the Codespaces Ports panel — it must be **Public**.

To confirm from your laptop's browser directly, paste the forwarded 3000 URL followed by `/branches` — e.g. `https://your-codespace-3000.app.github.dev/branches`. If this returns JSON, the port is public and something else is wrong. If it returns a GitHub sign-in page, the port is still private.

### Green connection dot but requests fail with CORS errors

Rare, but if it happens: add this env var to the `postgrest` service in `docker-compose.yml`:

```yaml
PGRST_SERVER_CORS_ALLOWED_ORIGINS: "https://your-codespace-3001.app.github.dev"
```

Then `docker compose up -d` to restart.

### Changing `db/init.sql` and it doesn't seem to update

Postgres only runs init scripts on a *fresh* volume. To rerun after edits:

```bash
docker compose down -v    # -v drops the volume
docker compose up -d
```

## The four ideas worth taking away

**1. Tables, views, and functions become URLs automatically.**

| In the database | Over HTTP |
|---|---|
| `api.branches` (table) | `GET /branches` |
| `api.public_branches` (view) | `GET /public_branches` |
| `api.branches_near()` (function) | `POST /rpc/branches_near` |

**2. The querystring is the query.**

```
GET /branches?select=name,lat&lat=gt.52.5&order=lat.desc&limit=4
```

```sql
SELECT name, lat FROM api.branches
WHERE lat > 52.5 ORDER BY lat DESC LIMIT 4;
```

Nobody wrote an endpoint for that combination. It just works. Operators: `eq`, `gt`, `lt`, `gte`, `lte`, `like`, `in`, `is`.

**3. Authorization lives in Postgres, not in application code.**

`authenticator` is a deliberately powerless role — it can log in but owns no data access. Per request, PostgREST switches into `web_anon` (unauthenticated) or `web_user` (authenticated).

Row Level Security then decides what each role can see:

```sql
CREATE POLICY branches_anon_select ON api.branches
  FOR SELECT TO web_anon
  USING (is_active = true AND show_on_b2c = true);
```

Verified behaviour: `web_anon` sees **7 branches**, `web_user` sees all **10**. The three internal depots aren't filtered by the API — the database physically refuses to return them.

**4. Grants can be asymmetric, and that's useful.**

`web_anon` has `INSERT` on `api.enquiries` but no `SELECT`. Anyone can submit an enquiry; nobody can read them back. That's a public contact form, enforced at the database.

## Try these directly

```bash
# Filtering
curl "http://localhost:3000/branches?postcode=eq.LE1%201AA"

# Composing filters, ordering, and column selection
curl "http://localhost:3000/branches?select=name,lat&lat=gt.52.5&order=lat.desc"

# The RPC — haversine distance
curl -X POST http://localhost:3000/rpc/branches_near \
  -H "Content-Type: application/json" \
  -d '{"lat": 52.6369, "lon": -1.1398, "radius_miles": 40}'

# Insert works
curl -X POST http://localhost:3000/enquiries \
  -H "Content-Type: application/json" \
  -d '{"branch_id":1,"name":"Ada","email":"ada@example.com","message":"M6 bolts?"}'

# Reading back does not (401 — by design)
curl http://localhost:3000/enquiries

# PostgREST self-documents: OpenAPI spec of every generated endpoint
curl http://localhost:3000
```

## Where this fits, honestly

**Good fit:** internal tools, admin panels, prototypes, read-heavy APIs — anywhere the data model *is* the API and you want to skip CRUD boilerplate.

**Poor fit:** complex orchestration, heavy business logic, workflows that don't map cleanly onto database operations. You *can* push that into PL/pgSQL functions, but significant business logic in PL/pgSQL is usually worse than in TypeScript — harder to test, debug, and hire for.

## What isn't in here yet

- JWT authentication. PostgREST reads a `role` claim from a JWT and switches into that Postgres role for the request — which is how `web_user` gets used in practice. Set `PGRST_JWT_SECRET` and send `Authorization: Bearer <token>`.
- A UI for the `marketConfigurations` half of the schema.
- Persistent Postgres volume for Codespaces. Right now if you rebuild the container, `init.sql` re-runs from scratch (fine for demos, annoying if you want data continuity).
